# v0.2 Retrieval Freeze

## Purpose

v0.2 should freeze the retrieval architecture as a measurable baseline.
The project already has enough retrieval layers. The risk now is uncontrolled
complexity, not insufficient technique.

This phase turns the current system into a stable reference point before
productization starts.

## Current Architecture To Freeze

The v0.2 retrieval stack is:

```text
query
  -> optional query decomposition
  -> dense vector search
  -> BM25 keyword search
  -> Tapestry graph search / PPR spreading activation
  -> RRF merge
  -> optional Muse routing boost/filter
  -> temporal rerank when time expressions are present
  -> ACT-R activation rerank
  -> optional parent-section expansion
```

This is the default architecture unless evaluation proves a specific component
should be disabled or reweighted.

## Scope

Allowed:

- fix bugs in the existing retrieval path
- improve evaluation hygiene
- tune existing weights with before/after reports
- document known tradeoffs
- make current generated state reproducible
- add small tests around existing behavior

Not allowed by default:

- new retrieval engines
- new graph algorithms
- new rerankers
- new LLM-heavy enrichment passes
- broad schema rewrites

Any exception needs a short note describing the failed evaluation or concrete
user-facing bug that justifies it.

## Known Tradeoffs

### Muse Routing

Muse routing improves top-rank usefulness but can reduce long-tail recall.
The current product decision is:

- optimize for `Recall@1`, `Recall@5`, and `MRR`
- accept some `Recall@10` loss
- keep hard routing optional
- default to soft/adaptive routing when enabled

This should be documented as a user-experience decision, not hidden as a
technical accident.

### ACT-R Rerank

ACT-R makes memory feel personalized by lifting recently or frequently used
items. It can also reinforce old search habits.

v0.2 should keep ACT-R, but evaluation reports should preserve raw retrieval
metrics where possible so ACT-R does not hide regressions.

### Graph Dependence On Enrichment

Tapestry quality depends on Oracle extraction quality. If entities are wrong,
graph retrieval can amplify the error.

v0.2 should rely on Aletheia and Augury to correct errors instead of trying to
make enrichment perfect.

## Validation Commands

Use the project virtualenv:

```bash
workon personal-memory
```

Core health checks:

```bash
python Personal_Brain_DB/00_System/mneme_weight.py --stats
python Personal_Brain_DB/00_System/tapestry.py --stats
python Personal_Brain_DB/00_System/slumber.py --stats
```

Retrieval regression check:

```bash
cd Personal_Brain_DB/00_System
make eval-ci
```

If `make eval-ci` is too slow, run a smaller local check and mark it as
non-release validation:

```bash
python benchmark/retrieval_eval.py --config full --n 100 --hygiene --diff
```

## Augury Golden Set

The golden set should represent actual use, not artificial benchmark queries.

Minimum target:

- 5 recall questions for journal memories
- 5 recall questions for AI chat memories
- 5 temporal questions
- 5 person/place/event association questions
- 5 negative or ambiguous questions where abstention matters

Each item should include:

- natural query
- expected memory path or acceptable path set
- reason this question matters
- category: recall, temporal, entity, correction, abstention

## Exit Criteria

v0.2 is done when:

- retrieval defaults are frozen
- evaluation baseline is reproducible
- TODO state matches reality
- current private vault generated changes are accounted for
- no new retrieval feature is pending as a blocker

After this point, retrieval changes should be treated as regressible product
changes, not exploratory implementation.
