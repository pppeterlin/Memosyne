# v0.2 檢索凍結

## 目的

v0.2 的目標是把目前檢索架構凍結為可測量、可重現的 baseline。

目前系統已經有足夠多的檢索層。下一個主要風險不是技巧不足，而是複雜度失控。因此本階段的重點是收斂、驗證、記錄取捨，而不是再增加新模組。
文檔與示例資料規範遵守 workspace `AGENTS.md`。

## 要凍結的目前架構

v0.2 檢索路徑：

```text
query
  -> optional query decomposition
  -> dense vector search
  -> BM25 keyword search
  -> Tapestry graph search / PPR spreading activation
  -> RRF merge
  -> optional Muse routing boost/filter
  -> temporal rerank when time expressions are present
  -> ACT-R activation rerank
  -> optional parent-section expansion
```

除非評估證明某個元件需要停用或調權，否則這就是 v0.2 預設架構。

## Scope

允許：

- 修正既有檢索路徑 bug
- 改善 evaluation hygiene
- 在有 before/after report 的前提下調整既有權重
- 記錄已知 tradeoff
- 讓目前生成狀態可重現
- 為既有行為補小型測試

預設延後：

- 新檢索引擎
- 新 graph algorithm
- 新 reranker
- 新 LLM-heavy enrichment pass
- 大型 schema rewrite

若要例外，必須先留下短說明：目前哪個評估失敗或哪個使用者可見 bug 需要這個變更。

## 已知取捨

### Muse Routing

Muse routing 能改善高順位結果，但可能降低長尾召回。

目前產品決策：

- 優先看 `Recall@1`、`Recall@5`、`MRR`
- 接受部分 `Recall@10` 代價
- hard routing 保持可選
- 預設偏向 soft/adaptive routing

這應被記錄為使用體驗決策，而不是隱藏在實作裡。

### ACT-R Rerank

ACT-R 會提高近期或高頻使用記憶的排序，使系統更個人化；同時也可能放大既有搜尋習慣。

v0.2 保留 ACT-R，但評估報告應盡量保留 raw retrieval 指標，避免 ACT-R 掩蓋底層召回退步。

### Graph 依賴 Enrichment 品質

Tapestry 的品質依賴 Oracle 抽取品質。若 entity extraction 有誤，graph retrieval 可能放大錯誤。

v0.2 不追求 enrichment 一次完美，而是依靠 Aletheia、Augury 與可回溯修正流程管理錯誤。

## Augury Golden Set

golden set 應代表真實使用型態，但問題內容必須合成或脫敏，不能使用私有記憶原句。

最低目標：

- 5 題 journal recall
- 5 題 AI chat recall
- 5 題 temporal query
- 5 題 person/place/event association
- 5 題 negative 或 ambiguous query，用於測 abstention

每題包含：

- natural query：合成查詢
- expected path 或 acceptable path set：可用 sample path
- reason：這類問題為什麼重要
- category：recall、temporal、entity、correction、abstention
