# v0.4 Open-Source Release

## Purpose

v0.4 should prepare Memosyne for public release without leaking private data
or requiring readers to understand the author's personal vault.

The release should communicate a clear product:

> Local-first personal memory infrastructure for AI agents, with ingest,
> hybrid retrieval, graph memory, correction, evaluation, and MCP integration.

## Public Positioning

Do not lead with a list of research techniques.

Lead with the user problem:

- AI agents do not know the user's long-term personal context.
- Personal context is scattered across journals, chats, notes, and profiles.
- A local memory layer should be searchable, correctable, auditable, and
  agent-accessible.

Then explain the architecture as the implementation of that product need.

## Release Boundary

Public repo should include:

- source code
- docs
- sample vault
- sample evaluation set
- install instructions
- MCP setup guide
- privacy model
- contribution guide
- license

Public repo must not include:

- private `_vault` contents
- API keys
- personal benchmark reports
- private cache files
- `.env` with real values
- generated DBs derived from private data
- local machine paths except examples

## Sample Vault

The sample vault is mandatory. Without it, the project is hard to evaluate.

Minimum sample structure:

```text
sample_vault/
  10_Profile/
    bio.md
    career.md
    preferences.md
  20_AI_Chats/
    Gemini/
      sample_chat.md
  30_Journal/
    2026/
      260101.md
      260107.md
  40_Projects/
    sample_project.md
  50_Knowledge/
    sample_note.md
```

The sample should demonstrate:

- person/place/event extraction
- temporal search
- AI chat vs journal retrieval
- one correction flow
- one graph association query
- one negative/abstention query

Use synthetic but realistic content. Do not anonymize real private data unless
it is heavily rewritten enough to be non-sensitive.

## OSS Documentation Set

Required public docs:

- `README.md`: product overview, quickstart, architecture, limitations
- `docs/quickstart.md`: clean-machine setup
- `docs/privacy.md`: local/cloud boundaries and data handling
- `docs/configuration.md`: paths, providers, model settings
- `docs/mcp.md`: Claude Desktop / Cursor integration
- `docs/evaluation.md`: Eternal Mirror and Augury
- `docs/development.md`: repo layout and contribution workflow
- `docs/sample_vault.md`: sample data walkthrough

Chinese docs can remain, but the public default should be concise English if
the target is broader OSS adoption.

## Packaging Target

Target install path:

```bash
pip install memosyne
```

Acceptable pre-release path:

```bash
git clone https://github.com/<owner>/memosyne
cd memosyne
pip install -e .
memosyne init --sample
```

Package metadata should define:

- Python version range
- console script entry point
- dependencies
- optional extras for local LLM, cloud LLM, eval, and dev

Example extras:

```text
memosyne[local]
memosyne[cloud]
memosyne[eval]
memosyne[dev]
```

## Privacy Requirements

`docs/privacy.md` should answer:

- What stays local?
- What can be sent to cloud LLMs?
- Which commands call LLM providers?
- Which files are generated from private data?
- Which paths should be gitignored?
- How do dry-run correction tools work?
- How can a user delete or rebuild derived indexes?

Default behavior should favor local operation or explicit opt-in for cloud
calls.

## Evaluation Story

Memosyne has a strong differentiator: retrieval evaluation is built in.

Public docs should explain two evaluation modes:

- Eternal Mirror: self-supervised round-trip retrieval using generated HyQE
  questions.
- Augury: human-authored golden questions for real user intent.

The sample vault should ship with a tiny evaluation set so contributors can
run:

```bash
memosyne eval --sample
```

## Security And Cleanup Checklist

Before public release:

- rotate any key that ever appeared in the repo
- remove private key files from history if needed
- verify `.gitignore` covers private vaults and generated DBs
- run a secret scanner
- verify sample data is synthetic
- check submodules do not point to private memory content
- check Docker image does not copy private files
- remove machine-specific absolute paths from public docs

## Open-Source Scope Control

Avoid promising:

- hosted sync
- multi-user collaboration
- mobile apps
- perfect memory extraction
- medical/legal-grade correctness
- universal agent behavior

Explicitly state limitations:

- retrieval quality depends on source material and enrichment quality
- graph memory can amplify extraction errors
- local model setup can be heavy
- evaluation metrics are useful but not a substitute for user review

## Exit Criteria

v0.4 is done when:

- clean install works
- sample vault demo works
- public docs explain the system without private context
- privacy boundary is explicit
- no private data or keys are included
- at least one MCP client setup path is verified
- evaluation can run on sample data

At this point Memosyne can be promoted as an open-source personal memory
infrastructure project rather than an internal research repo.
