# v0.3 使用產品化

## 目的

v0.3 的目標是讓 Memosyne 能被穩定地日常操作。

目前系統能力完整，但入口偏腳本化。產品化不是新增功能，而是建立單一清楚的 command surface、明確前置條件、可預期維護流程與可理解的失敗訊息。
文檔與示例資料規範遵守 workspace `AGENTS.md`。

## 產品目標

技術型使用者應能在不先閱讀內部實作的情況下完成：

1. 建立 vault
2. 入庫檔案
3. 重建索引
4. 搜尋記憶
5. 檢查健康狀態
6. 啟動 MCP server
7. 安全修正記憶

## 建議 Command Surface

目標 CLI：

```bash
memosyne init
memosyne ingest
memosyne search "測試查詢"
memosyne rebuild
memosyne health
memosyne mcp
memosyne correct ...
memosyne slumber --stats
```

第一版已採取薄包裝策略：repo root 的 `memosyne.py` 包裝既有腳本，不立即大重構。安裝後透過 `pyproject.toml` 暴露 `memosyne` console script。

目前可用：

```bash
python3 memosyne.py init
python3 memosyne.py ingest
python3 memosyne.py search "測試查詢"
python3 memosyne.py rebuild
python3 memosyne.py health
python3 memosyne.py mcp --check
python3 memosyne.py chronicle --stats
```

或：

```bash
pip install -e .
memosyne health
```

## Operational Health Checks

`memosyne health` 應檢查：

- Python 版本是否支援
- 依賴套件是否可 import
- vault path 是否存在
- Chroma DB 是否存在或可建立
- BM25 index 是否存在
- Tapestry DB 是否存在
- Chronicle DB 是否存在
- embedding model 是否可從 cache 載入，或是否需要網路
- enrichment 所需 LLM provider 是否可用
- MCP server 是否可正常 import
- secret 是否誤放在公開路徑

輸出應能直接指示下一步：

```text
[ok] Python 3.11
[ok] Kuzu import
[warn] embedding model not cached; first search may need network access
[fail] missing Personal_Brain_DB/_vault
```

## Error Handling 要求

常見失敗應說明下一步，而不是只丟 raw traceback：

- virtualenv 未啟用
- Python 版本錯誤
- 缺少 `kuzu`
- Hugging Face model 未 cache 且無網路
- Ollama 未啟動
- cloud LLM key 缺失
- Aletheia 修正後索引過期
- vault submodule dirty

## Artifact Strategy

v0.3 的 artifact 原則是把「可攜帶的私有狀態」與「可重建的本機快取」分開。

### 路徑原則

- canonical private artifacts 預設放在 `Personal_Brain_DB/_vault/`
- `Personal_Brain_DB/00_System/` 中既有 symlink shim 繼續相容舊腳本
- 大型本機快取留在 `00_System/` 並由 `.gitignore` 排除
- 可用 `MEMOSYNE_VAULT_DIR` 或 `MEMOSYNE_ARTIFACT_DIR` 覆寫 artifact 根目錄

### Artifact 分類

| Artifact | 類型 | 策略 |
|---|---|---|
| `chronicle.jsonl` | append-only source log | Chronicle 真相來源，一行一個 access event |
| `chronicle.db` | derived SQLite cache | ACT-R scoring / stats 快取，可由 JSONL 重建 |
| `bm25_index.pkl` | derived retrieval index | 私有 vault chunk 的 sparse index，可重建 |
| `contextual_cache.json` | LLM cache | The Illumination 產物，避免重複 LLM 呼叫 |
| `hyqe_cache.json` | LLM cache | The Triple Echo 產物，避免重複 LLM 呼叫 |
| `tapestry_db/` | derived graph store | The Tapestry 圖資料庫，可由 enriched vault 重建 |
| `muse_centroids.json` | derived routing data | Muse router centroid，可重建 |
| `chroma_db/` | local runtime cache | 體積較大，保留在 `00_System/`，不進公開 repo |

### Chronicle JSONL

Chronicle 從 v0.3 起採取雙層儲存：

1. `chronicle.jsonl`：append-only event log，適合 diff、備份、同步與未來 audit。
2. `chronicle.db`：SQLite 查詢快取，供 `compute_activation()` 快速計算 ACT-R 分數。

日常搜尋會同時寫入 SQLite 與 JSONL。若 SQLite 損壞或需要重建，可從 JSONL 重新生成：

```bash
python3 Personal_Brain_DB/00_System/mneme_weight.py --rebuild-db-from-jsonl
```

若升級前已有 `chronicle.db`，可一次性匯出既有事件：

```bash
python3 Personal_Brain_DB/00_System/mneme_weight.py --export-jsonl --replace-jsonl
```

這個策略讓 The Chronicle 保持可讀、可攜、可重建；SQLite 不再承擔唯一真相來源。

## MCP 產品化

MCP server 應穩定暴露：

- `search_memory`
- `read_file`
- `get_profile`
- `list_journals`
- `get_memory_health`
- `optimize_memory`
- `query_memory_at_time`
- `get_entity_timeline`
- Aletheia correction tools
- `memosyne_guide`

v0.3 至少應標準化以下行為：

- write/correction tools 預設 dry-run
- 每個 retrieval result 都包含 path
- 錯誤訊息清楚
- 不做 silent writes

完整 schema 統一可延後，但公開行為不應頻繁變動。

## 驗證

v0.3 release 前：

```bash
source .venv/bin/activate
memosyne health
memosyne search "測試查詢"
memosyne slumber --stats
memosyne mcp --check
```

若 `memosyne` 尚未實作，需先以既有命令驗證並在文檔中標示：

```bash
python Personal_Brain_DB/00_System/vectorize.py --query "測試查詢" --top 3
python Personal_Brain_DB/00_System/slumber.py --stats
python Personal_Brain_DB/00_System/mcp_server.py
```
