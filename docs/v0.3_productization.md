# v0.3 Productized Usage

## Purpose

v0.3 should make Memosyne practical to operate every day.

The current system is powerful but script-heavy. Productization means one
clear command surface, explicit prerequisites, predictable maintenance, and
understandable failure modes.

## Product Goal

A technically comfortable user should be able to:

1. create a vault
2. ingest files
3. rebuild indexes
4. search memories
5. inspect health
6. run the MCP server
7. correct a memory safely

without needing to understand the internals first.

## Proposed Command Surface

Target CLI:

```bash
memosyne init
memosyne ingest
memosyne search "深圳工作"
memosyne rebuild
memosyne health
memosyne mcp
memosyne correct ...
memosyne slumber --stats
```

Initial implementation can wrap existing scripts rather than refactor them.
The important point is to establish the public interface.

## Scope

Allowed:

- package entry point
- CLI wrappers around existing scripts
- health checks
- clearer error messages
- environment validation
- documentation cleanup
- sample configuration files
- small refactors that reduce duplicated path handling

Deferred:

- replacing ChromaDB
- replacing Kuzu
- full UI
- multi-user hosting
- remote sync
- new retrieval algorithms

## Operational Health Checks

`memosyne health` should check:

- Python version is supported
- required packages import
- vault path exists
- Chroma DB exists or can be created
- BM25 index exists
- Tapestry DB exists
- Chronicle DB exists
- embedding model can load from cache or network is available
- configured LLM provider is available when enrichment is requested
- MCP server imports cleanly
- private keys are not accidentally inside public paths

The command should report actionable status:

```text
[ok] Python 3.11
[ok] Kuzu import
[warn] embedding model not cached; first search may download from Hugging Face
[fail] missing Personal_Brain_DB/_vault
```

## Documentation Set

v0.3 should contain:

- `README.md`: high-level value and quickstart
- `README.zh.md`: Chinese quickstart and conceptual guide
- `docs/operations.md`: daily commands and troubleshooting
- `docs/configuration.md`: paths, env vars, LLM providers
- `docs/mcp.md`: Claude Desktop / Cursor setup
- `docs/correction.md`: Aletheia safe correction flow
- `docs/evaluation.md`: Eternal Mirror and Augury usage

## Error Handling Requirements

Common failures should explain the next action:

- missing virtualenv
- wrong Python version
- missing `kuzu`
- Hugging Face model not cached and network unavailable
- Ollama not running
- cloud LLM key missing
- stale index after Aletheia correction
- dirty vault submodule

Avoid raw tracebacks for expected operational failures.

## Configuration Model

Target config file:

```toml
[paths]
vault = "Personal_Brain_DB/_vault"
spring = "spring"
system = "Personal_Brain_DB/00_System"

[models]
embedding = "paraphrase-multilingual-MiniLM-L12-v2"
enrichment = "gemma4:26b"
aggregation = "proxy:claude-opus-4-6"

[retrieval]
default_top_k = 5
auto_route = true
muse_mode = "soft"
```

Existing defaults can remain in code for v0.3, but docs should describe where
configuration is heading.

## MCP Productization

The MCP server should expose a stable set of tools:

- `search_memory`
- `read_file`
- `get_profile`
- `list_journals`
- `get_memory_health`
- `optimize_memory`
- `query_memory_at_time`
- `get_entity_timeline`
- Aletheia correction tools
- `memosyne_guide`

v0.3 should standardize return style enough that agents can rely on it:

- dry-run defaults for write/correction tools
- path included in every retrieval result
- clear error messages
- no silent writes

Full schema consistency can be deferred, but the public behavior should stop
changing casually.

## Validation

Before v0.3 release:

```bash
workon personal-memory
memosyne health
memosyne search "深圳工作"
memosyne slumber --stats
memosyne mcp --check
```

If `memosyne` is not implemented yet, equivalent existing commands must be
documented and pass:

```bash
python Personal_Brain_DB/00_System/vectorize.py --query "深圳工作" --top 3
python Personal_Brain_DB/00_System/slumber.py --stats
python Personal_Brain_DB/00_System/mcp_server.py
```

## Exit Criteria

v0.3 is done when:

- the normal user journey is documented end to end
- one command surface exists for common operations
- health checks catch known environment problems
- correction and rebuild flows are documented
- MCP setup can be completed from docs without reading source code
- retrieval behavior remains within v0.2 baseline tolerance
