# 專案藍圖：基於 MCP 協議的仿生個人記憶庫 (Memory Base)

## 1. 核心定位 (Core Vision)
本專案並非打造一個智能體（Agent），而是打造一個「記憶基礎設施層」。

* **非代理化 (Non-agentic)**：記憶庫是一個被動的、可被調用的知識基座。
* **跨代理調用**：透過 MCP (Model Context Protocol) 協議，讓不同的 Agent (如 IDE Agent, CLI, LLM API) 能夠統一調用使用者的個人記憶。
* **仿生架構**：模擬人類記憶的層級化、關聯化與遺忘機制，提升資訊的檢索信噪比。

## 2. 技術棧建議 (Technical Stack)
* **通訊協議**：MCP (Model Context Protocol)
* **存儲層**：
  * **向量存儲 (Vector Store)**：SQLite-vec 或 ChromaDB（儲存檔案切片與語境化 Embedding）。
  * **拓樸圖譜 (Graph Store)**：NetworkX (Python) 或 SQLite-Graph（儲存實體間的關係）。
  * **持久化存儲**：本地 Markdown 檔案庫（保持人類可讀性與版本控制便利性）。

## 3. 功能模塊設計 (Functional Modules)

### A. 語境化編碼 (Contextual Ingestion Pipeline)
* **目的**：解決 RAG 切片斷章取義的問題，模擬人類「帶有背景的感知」。
* **實作細節**：在進行文本切片（Chunking）前，先利用 LLM 生成該片段的「全局摘要」。
* **存儲格式**：將摘要與切片合併後進行 Embedding。
* **例**：`[文件類型: 履歷] + [這段內容關於: 2023 年在 X 公司的技術棧] + [原始文本內容]`

### B. 階層式記憶架構 (Hierarchical Memory)
模擬人類長期記憶的二元結構：
* **情節記憶 (Episodic Memory)**：儲存具體對話、日記、互動日誌。具有時間戳與場景標籤。
* **語義記憶 (Semantic Memory)**：儲存經過蒸餾的事實、偏好與概念。以「向量＋拓樸關係圖」呈現。
  * **節點 (Nodes)**：實體（如：Python, 專案 A, 使用者偏好）。
  * **邊 (Edges)**：關係（如：熟悉, 參與, 討厭）。

### C. 關聯激發檢索 (Spreading Activation Retrieval)
* **目的**：超越單純相似度，實現跨文檔的邏輯聯想。
* **種子定位**：透過向量搜尋找到與 Query 最相似的 Top-K 節點。
* **拓樸擴散**：以這些節點為起點，在圖譜中運行 Personalized PageRank (PPR) 或傳播激發演算法，挖掘二度、三度關聯。
* **ACT-R 權重重排**：利用 ACT-R 基礎激活公式計算記憶權重：

$$B_i = \ln(\sum_{k=1}^n t_k^{-d})$$

> （$n$ 為使用頻率，$t_k$ 為距離上次調用的時間，$d$ 為衰減因子，建議設為 0.5）。

### D. 睡眠鞏固與優化 (Consolidation & Pruning)
* **目的**：非同步處理，保持記憶庫的整潔與進化。
* **反射機制 (Reflection)**：當新增記憶達到門檻時，啟動後台任務進行「蒸餾」。將多則「情節記憶」合併為一則穩定的「語義記憶」。
* **衝突處理**：若新入庫資料與舊有記憶衝突（如：工作地址變更），自動更新圖譜節點權重或標記舊記憶為「過時」。
* **戰略遺忘**：自動清理重要性評分極低且長期未被調用的切片，降低干擾。

## 4. MCP Tool 定義 (API for Agents)
Claude Code 需實作以下 MCP 工具介面：

* `add_to_memory(content, source_type, metadata)`: 將檔案、對話或日誌寫入系統。
* `query_memory(query, k)`: 返回結合向量相似度與圖譜聯想的混合檢索結果。
* `get_user_profile()`: 從「語義記憶」中直接提取使用者的個人偏好與關鍵背景資訊。
* `consolidate_memory()`: 手動或自動觸發「睡眠循環」，進行知識整理與去重。
