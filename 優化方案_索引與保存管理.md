## 1. 索引層優化（Indexing Strategies）

### 1.1 多視角索引：The Triple Echo

**動機**：單一 chunk 只生成一個 embedding，無法涵蓋使用者用各種口吻提問的可能。

**作法**：為每個 chunk 產生三種視角，全部嵌入同一個 collection（用 metadata 標記視角類型），查詢時 max-pooling：

```
view_raw        = 原始段落
view_summary    = Oracle 已產出的 contextual summary（現已有）
view_hyqe       = Oracle 從 chunk 反推「這段可以回答哪 3–5 個問題」
```

**HyQE 範例**（Hypothetical Question Embedding）：
```yaml
# 自動附加到 YAML
hyqe_questions:
  - "我在 Tokyo 的那段時間認識了誰？"
  - "friend-A 的 project 是關於什麼？"
  - "2023 年夏天我的情緒狀態如何？"
```

**儀式命名**：**The Triple Echo**（三重回聲 —— 一段記憶，三種召喚方式）

**成本**：低（LLM 呼叫 ×3，但可延用 Illumination 的 cache 機制）
**與現有架構相容度**：完美 —— 直接新增 metadata.view ∈ {raw, summary, hyqe}，搜尋時以最高分那個為該 chunk 代表分數。

**參考**：[HyDE (arxiv 2212.10496)](https://arxiv.org/abs/2212.10496)、Anthropic Contextual Retrieval cookbook

---

### 1.2 親子切片：Small-to-Big（The Thread and the Tapestry）

**動機**：embedding 在小 chunk 上最準，但 LLM 讀大 chunk 才能答對。

**作法**：
- 索引層：切 ~200 token 小片段（當前 chunker 已近此大小，微調即可）
- 儲存層：每個小片段的 metadata 記錄 `parent_section`（同一 H2 標題下的內容）與 `parent_doc`
- 檢索層：搜到小片段 → 回傳整個 parent_section（或在 token 預算內擴張）

**實作重點**：ingest.py 切片時，同時寫入：
```python
metadata = {
    "chunk_id": "...",
    "parent_section_id": "...",   # 同一 H2 下的所有 chunk 共用
    "parent_doc_id": "...",
    "sibling_order": 3,
}
```

**成本**：低（只需調整 chunker 與 search 回傳邏輯）
**儀式命名**：The Thread（索引用的絲線） and The Tapestry（呈現的織品）

---

### 1.3 命題化切片（Propositional Chunking）— 可選進階

**動機**：日記/對話段落長短不均，固定大小切片會把一個完整想法切兩半。

**作法**：用 LLM 將每段日記拆成「獨立陳述句列表」，每句一個向量。典型例：
> 原文：「那天跟 A 見面，他還在 Tokyo 做那個機器學習的 project，我覺得他有點疲倦但很專注。」
>
> 命題化：
> - `2025-06-04 與 A 見面`
> - `A 當時在 Tokyo`
> - `A 在做機器學習 project`
> - `A 顯得疲倦但專注`

**成本**：中（每篇日記 LLM ×1，可在 Slumber 階段離線處理）
**建議**：先不做 —— 跟 HyQE 重疊度高，優先做 1.1。若 1.1 做完發現還有召回問題再加。

## 2. 搜尋層優化（Retrieval Strategies）

### 2.1 繆思路由器：The Invocation of the Muses

**動機**：你已經把記憶按 9 位繆思分類（Clio=日記、Urania=知識、Calliope=對話…），查詢時卻沒利用這個分類。例如「我 2023 年的技術棧是什麼」顯然應該優先搜 Urania + 40_Projects，不該浪費分數在 Erato 的情感記憶上。

**作法**：在 query 進入三路檢索之前，先用輕量 LLM（或 embedding 相似度分類器）把 query 映射到 1–3 位繆思：

```python
def invoke_muses(query: str) -> list[str]:
    """根據 query 決定召喚哪幾位繆思"""
    # gemma3:4b 或 sentence-transformer 分類
    # 輸出: ["Clio", "Terpsichore"]  → 搜 30_Journal + 40_Projects
```

然後：
- **硬篩選**（查詢有明確領域詞）：只搜該繆思的子集
- **軟加權**（一般查詢）：對該繆思的結果 ×1.3

**成本**：低（分類器可用已有的 embedding model）
**儀式命名**：**The Invocation** —「神諭召喚對應的繆思降臨」

---

### 2.2 查詢分解與時間擴展：The Unfolding Question

**動機**：「2023 年夏天我和 friend-A 在 Tokyo 做了什麼」是一個複合 query，包含 (時間, 人, 地點, 動作) 四個約束。LongMemEval 論文顯示，分解後分頭搜尋再合併的召回顯著優於整句搜尋。

**作法**：
1. **時間錨點抽取**：regex / LLM 抽出「2023 年夏天」→ 過濾 `YYYY-MM` ∈ {2023-06, 07, 08}
2. **實體抽取**：friend-A, Tokyo → 直接當 Tapestry seed 節點
3. **分頭檢索**：每個子約束各自走一次 RRF，最後再 RRF 合併
4. **時間感知重排**：命中的記憶日期 vs. 查詢時間錨點越近 → bonus

**成本**：中（多一次 LLM 分解 + 多幾輪檢索）
**建議先做時間擴展**（regex 就能處理 80%），LLM 分解留給 v2

**參考**：[LongMemEval (arxiv 2410.10813)](https://arxiv.org/abs/2410.10813)

---

### 2.3 HippoRAG 2 升級：短語節點與段落節點共圖

**動機**：你目前的 PPR 只在實體圖（person/location/event）上擴散；HippoRAG 2 的做法是把「短語節點」（實體）和「段落節點」（記憶）放同一張圖，PPR 同時擴散過兩種節點。效果：找到 `Tokyo` → 擴散到 `friend-A` 節點 → 擴散到**所有提到 friend-A 的段落**（不只是直接提 Tokyo 的）。

你的 Tapestry 已經有 memory → person/location 的 mentions 邊，理論上半步就到位。需要的是：
- PPR seed 允許同時是 phrase node 和 passage node
- 擴散後回傳時，只取 passage node 的分數作為該 memory 的貢獻

**成本**：中（tapestry.py 的 PPR 調用處改寫，約 50–100 行）
**與 ACT-R 疊加**：PPR 分數 → RRF → ACT-R bonus，順序不變

**參考**：[HippoRAG 2 (arxiv 2502.14802)](https://arxiv.org/abs/2502.14802)

---

### 2.4 ColBERT 晚期互動 —— 先別急

**評估**：ColBERT 對於「實體密集的短查詢」有明顯增益，但索引體積 ×3–5，對於個人記憶庫（規模幾千–幾萬段落）性價比低。**優先級：等 1.1 + 2.1 + 2.3 做完、評估證明 dense leg 是瓶頸再考慮**。

## 3. 時序與實體管理（Temporal & Entity Layer）

### 3.1 Bi-temporal Tapestry：The Two Rivers of Time

**動機**（最重要的結構性升級之一）：

目前 Tapestry 的邊只記錄 ingest_time（寫入時間），無法區分：
- **valid_time**：事實在世界上成立的時間（friend-A 在 Tokyo 的 2020-2024）
- **ingest_time**：你把這件事寫進記憶庫的時間（2024-03-15 那天的日記）

**後果**：
- 當 `friend-A 搬到 Osaka` 被記錄後，舊的 `friend-A 住 Tokyo` 應該被「失效」而非刪除
- 無法回答「我 2023 年當時以為 friend-A 住哪」vs「friend-A 實際住哪」

**作法**（直接抄 Graphiti 的做法）：
```python
# 每條邊變成
edge = {
    "src": "friend-A",
    "dst": "Tokyo",
    "type": "located_in",
    "t_valid_start": "2020-01-01",
    "t_valid_end": "2024-03-15",  # 新事實出現時設定，不刪除
    "t_ingested": "2024-03-20",
    "invalidated_by": "memory-id-xyz",  # 指向導致失效的記憶
}
```

**查詢時可以問**：
- "當前有效"：`t_valid_end IS NULL`
- "某時間點的世界狀態"：`t_valid_start ≤ ts AND (t_valid_end IS NULL OR t_valid_end > ts)`
- "我以前誤以為的事實"：`t_valid_end IS NOT NULL`

**成本**：中（Tapestry schema 變動 + Slumber 新增衝突偵測儀式）
**儀式命名**：**The Two Rivers** —「Mnemosyne 之泉流過兩種時間：事件的時間，與知曉的時間」

**參考**：[Zep / Graphiti (arxiv 2501.13956)](https://arxiv.org/abs/2501.13956) — 在 LongMemEval 上比 MemGPT 高 18.5%

---

### 3.2 Session / Episode 節點

**動機**：目前 memory 節點是 chunk 級別，但人腦的情節記憶以「一次事件」為單位。「2023-09-15 與 friend-A 在澀谷吃飯」是一個完整 episode，不應該拆成三個獨立 chunk。

**作法**：在 Tapestry 新增 `episode` 節點類型，一篇日記 → 一個或多個 episode 節點，episode → chunk 之間以 `composed_of` 邊連結。PPR 擴散時經過 episode 節點 → 同一 episode 的所有 chunk 都被激發。

**成本**：低（ingest.py 在切片後額外產 episode 節點即可）

## 4. 生命週期管理（Lifecycle Management）

### 4.1 實體正規化：The Naming of the Muses

**動機**：`friend-A` / `friend A` / `friend_a` / 真名 — 對 Tapestry 是 4 個不同節點，PPR 分數被稀釋。

**作法**（在 Slumber 新增第四儀式）：

```
The Naming Rite（命名儀式）
  1. 收集所有 person 節點 → 取名字做 normalize（lowercase / 去連字號）
  2. 對正規化後的字串做 embedding 相似度聚類（>0.85）
  3. LLM 二次確認：「這些是同一個人嗎？」
  4. 合併節點：邊重導、保留所有別名至 aliases[]
  5. 記錄 merge log，可 reversible
```

**Schema**：
```yaml
person:
  canonical_name: "friend-A"
  aliases: ["friend A", "A", "小 A"]
  merged_from: ["node-id-1", "node-id-2"]
  merged_at: "2026-04-21"
```

**成本**：中（一次性 LLM 成本，之後新增記憶時即時 alias 偵測）
**儀式命名**：**The Naming Rite** —「真名是神聖的 —— 亂序的呼喚終將歸於一」

---

### 4.2 衝突處理：The Ordeal of Contradiction

**動機**：新記憶「A 搬到 Osaka」與舊記憶「A 住 Tokyo」衝突時，現在只是兩個都存在，查詢結果會同時命中兩個。

**作法**（整合 Mem0 的 CRUD 模式 + Graphiti 的邊失效）：

```
The Ordeal（在 enrich 或 slumber 時觸發）
  1. 新 fact 入庫時，用 embedding + LLM 比對是否與既有 semantic fact 衝突
  2. LLM 輸出 operation ∈ {ADD, UPDATE, INVALIDATE, MERGE, NOOP}
  3. ADD        → 新增
     UPDATE     → 舊 fact 的 valid_end 設為新 fact 的 valid_start
     INVALIDATE → 舊 fact 標記失效（不刪），記錄原因
     MERGE      → 兩 fact 合併為更完整的版本
     NOOP       → 重複資訊，跳過
  4. 所有操作可 reversible（寫 operation log）
```

**成本**：中（每次 enrich 多一次 LLM call，但只對 personal_facts 做，量不大）

**參考**：[Mem0 (arxiv 2504.19413)](https://arxiv.org/abs/2504.19413) — 在 LOCOMO 上比 OpenAI memory 高 26%

---

### 4.3 Oracle 自我批判：Self-RAG on Enrichment

**動機**：Oracle 偶爾會「超譯」—— 原文沒講的也被寫進 enrichment。雖然 CLAUDE.md 已明訂 ground-truth 原則，但 prompt 無法 100% 保證。

**作法**：enrich.py 產出 YAML 後，做一次 self-critique：
```
input: [enriched YAML] + [原文 chunk]
prompt: 逐項檢查 YAML 中的事實是否能在原文中找到字面依據。
output: {
  "personal_facts[0]": {"supported": true, "evidence": "第 3 段：'A 住 Tokyo'"},
  "entities.people[2]": {"supported": false, "reason": "原文未提此人，疑似幻想"},
  ...
}
```
不支持的項目自動剔除或標記 `needs_review: true`。

**成本**：中（每篇多一次 LLM call），可批次處理降低
**儀式命名**：**The Mirror of Truth** —「神諭說出之後，再次映照自身」

**參考**：[Self-RAG (arxiv 2310.11511)](https://arxiv.org/abs/2310.11511)

---

### 4.4 A-MEM 記憶演化：共鳴的更新

**動機**：你的 Hebbian 儀式只強化「共現邊」，但 A-MEM 更進一步 —— **新記憶可以回去修改舊記憶的屬性**。

**例子**：2025 年寫「A 是我大學同學」；2026 年寫「A 是我公司同事」。A-MEM 的做法：新增 2026 記憶時，觸發 Oracle 檢查相關舊記憶的 tags，可能在 2025 的記憶加上 tag `future_colleague`。

**作法**（作為 Slumber 的第五儀式）：
```
The Resonance（共鳴儀式）
  1. 新記憶的 entities 跟哪些舊記憶重疊？
  2. 對重疊 >= 2 的老記憶，讓 Oracle 重新產一次 enrichment（只增不減）
  3. 版本化：舊 enrichment 保留在 history[]，可追溯
```

**成本**：中（需控制頻率，避免每次 ingest 都觸發大量重算）
**建議**：只在 Slumber 排程時執行，不要即時做

**參考**：[A-MEM (arxiv 2502.12110)](https://arxiv.org/abs/2502.12110)

## 5. 評估機制：The Augury Benchmark

**這是最重要的一項** —— 沒有評估基準，前面所有優化都無法驗證是否真的改善。

### 5.1 黃金集建立

**作法**：
1. 手動為 9 位繆思的每個領域各建 10–20 個 query，標註預期命中的檔案
2. 格式：
```yaml
# Personal_Brain_DB/00_System/benchmark/golden_set.yaml
- id: q001
  muse: Clio
  query: "我 2023 年在 Tokyo 見過哪些人"
  expected_files:
    - "30_Journal/2023/230615.md"
    - "30_Journal/2023/230720.md"
  type: multi_hop  # 或 factual, temporal, reasoning
```
3. 月度更新（隨 Vault 成長）

### 5.2 指標

- **Recall@K**：expected_files 有多少在前 K 筆
- **MRR**：第一個命中的倒數排名
- **Precision@5**：前 5 筆中有多少是 expected
- **多跳完整率**：multi_hop 類型下，所有 expected_files 都命中的比例
- **時間正確率**：temporal 類型下，回傳的日期範圍正確比例

### 5.3 整合到工程流程

```bash
# 新儀式：augury_benchmark.py
python3 00_System/augury_benchmark.py --run
# → 輸出 reports/2026-04-21_retrieval_report.md
# → 對比上次結果，顯示各指標變化
```

每次改 search/vectorize/tapestry 的邏輯前後都跑一次，CI 化更佳。

### 5.4 可考慮移植的公開 benchmark

- **[LongMemEval](https://arxiv.org/abs/2410.10813)**：500 Qs / 5 種能力 / 對話記憶 —— 最接近你的使用場景
- **LOCOMO**：長對話記憶，Mem0 論文使用
- **RAGAS**：[docs.ragas.io](https://docs.ragas.io) —— 通用 faithfulness / relevancy 指標

**儀式命名**：**The Augury** —「定期占卜 Vault 的健康狀態」（CLAUDE.md 已保留此名）
